## Context

The vault's own docs (CLAUDE.md, vault_config.yaml, MAD_MAX_STATUS.md, Claude_on_Claude/docs/ARCHITECTURE.md, and ~10 knowledge entries) described a 3-vault ecosystem:

1. Claude_Vault — AI operational layer
2. Local-Network-Hub — human/AI coordination
3. Force Multiplication v1 — human knowledge base (read for RAG)

This was wrong on two axes:

- **FM v1 was never the right home for retrieval.** It is a legacy human notebook (1,600+ files, no frontmatter schema, unstructured). Agents were told to ingest it; nothing was ingested (MAD_MAX_STATUS.md still says "NEXT PRIORITY" for AnythingLLM ingestion as of 2026-04-17).
- **The system was missing a monitoring layer.** Command_Vault was built as a separate vault to watch all the others but never referenced anywhere in Claude_Vault's docs.

Result: tasks (TSK-0001, TSK-0009) were scoped against the wrong target, the briefing contained FM v1 context agents couldn't use, and scaffolding was built for pipes that pointed nowhere.

## Alternatives

1. **Keep 3-vault system, repurpose FM v1 as RAG source.** Rejected — FM v1 is unstructured human material; retrofitting schema across 1,600+ files is a project of its own. Leaving it in the RAG role also means the vault system depends on an external human notebook that changes independently.

2. **2-vault system (Claude_Vault + Local-Network-Hub), fold monitoring and RAG into Claude_Vault.** Rejected — violates single-purpose-per-vault. Claude_Vault is already overloaded (it contains llm-orchestrator, Claude_on_Claude, n8n-workflows, and the AI ops layer) and that overload is why nothing functions.

3. **4-vault system: Command_Vault monitors, Local-Network-Hub orchestrates, Claude_Vault is AI ops, RAG_Vault is retrieval.** Accepted. Separates concerns, gives each role a dedicated repo, removes FM v1 from the active system.

## Rationale

- **Command_Vault** (monitoring) is observability. It reads the other vaults, emits health + alerts, owns the Prometheus exporter + Grafana. Never writes operational data back.
- **Local-Network-Hub** (orchestration) is the REST API and routing layer. It owns n8n-workflows and the webhook endpoints. It's the nervous system that connects the other vaults to agents.
- **Claude_Vault** (AI ops) is the cognitive scaffold: decisions, knowledge, sessions, tasks. What this vault IS after extraction of llm-orchestrator and Claude_on_Claude.
- **RAG_Vault** (retrieval) is a structured knowledge corpus designed for ingestion into AnythingLLM / Smart Connections / local embeddings. Starts empty; content is migrated deliberately from Claude_Vault/knowledge/ and selectively from FM v1 as it gets reviewed.
- **FM v1** stays as a read-only reference. Not indexed, not linked, not in the briefing.
- **agentA-Z keyboard project** is iceboxed to `Documents\agentA-Z\` with its own repo. Not a vault, not blocking. _(Amended by [[DEC-0004]] 2026-05-28: the project was revived as `android-ai-keyboard-harness`; the `agentA-Z` folder was never created. The "not a vault" boundary stands; the icebox does not.)_

Each vault = its own git repo. No nested repos. This is the primary fix for the ghost-repo / drift-across-systems problem.
