## Decision

Use FlorisBoard as the IME foundation for agentA-Z Keyboard. Fork and extend with:
1. AI orchestration engine (background service reading skills.json)
2. Trigger detection in the NLP/spell-checker layer
3. CoT/ToT reasoning injected into the candidate suggestion bar
4. Voice integration via whisper-to-input pattern
5. Context engine (SQLite-vec + embeddings)

SwiftSlate (Accessibility Service approach) runs in parallel as the faster-to-prototype trigger layer during R&D, feeding learnings back into the FlorisBoard fork.

## Alternatives Considered

1. **AnySoftKeyboard** — Mature and stable but Java-based, older architecture. Extension system less flexible. Would require more work to add JSON-driven orchestration.

2. **Build from scratch** — Maximum control but massive effort. Android IME development is complex (InputMethodService, InputConnection, candidate views). Would take months before reaching feature parity with FlorisBoard's existing keyboard functionality.

3. **KeyboardGPT (Xposed)** — Fastest prototype (overlays any keyboard) but requires root/LSPosed. Limited to custom prompts — no deep integration with candidate bar, no voice, no template engine. Dead end for production distribution.

## Rationale

FlorisBoard balances development speed with extensibility. The existing JSON config system for themes/layouts provides a proven pattern we extend to AI skills. The spell-checker hook is the key technical enabler — it already fires on every input event, giving us the trigger mechanism for free. Kotlin + coroutines maps well to async LLM calls. Apache-2.0 means we can ship commercially if needed.

The two-track approach (SwiftSlate for fast R&D + FlorisBoard for product) de-risks the timeline: we get working triggers in days while building the full keyboard in weeks.
