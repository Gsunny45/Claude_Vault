## Context

Following the 2026-05-29 injection test suite ([[knowledge/KNW-0025]]), Mars set
two provider keys (Gemini + Groq) and authorized moving the project forward with
the recommended fixes. The suite had found the shipped `CteKeysActivity` launch
doc-comment was wrong and that injection silently no-ops when the Keys screen is
already open (singleTask, no `onNewIntent`).

## Decision

Applied to `android-ai-keyboard-harness` (separate repo, not a vault — boundary per
[[decisions/DEC-0004]]; recorded here as a pointer, code lives in the harness):

1. **Corrected the inject doc-comment** in `CteKeysActivity.kt` to the verified
   contract: explicit component **plus** `-a android.intent.action.VIEW -d
   "ui://hermetica-z/cte/keys"`, with the singleTask and whitespace-value caveats.
2. **Closed the singleTask re-inject gap**: extracted inject handling into
   `handleAdbInject(intent, keyVault): Boolean` and added an `onNewIntent` override
   that calls it, so a repeat `am start` re-injects instead of being dropped.
3. **Fixed the char-count log** to report `trimmed.length` (was logging the
   untrimmed value length).

Build `BUILD SUCCESSFUL`, installed on the Moto (ZY22G7NFLK). Re-inject fix verified
live: injecting into an already-open Keys screen now writes via `onNewIntent`
(`Key injected via ADB: __NEWINTENT_TEST__ (15 chars)`, count 4→5).

## Verified state (2026-05-29)

- KeyVault holds **2 user keys** (Gemini + Groq) + 2 Tink keyset entries.
- `triggers.json`: `gemini_1` (default), `gemini_2`, `groq` all `enabled: true`.
- `routing.json`: `default = gemini_1`, `standaloneMode: true` (bypasses dead local
  llama-server), cheap ladder `gemini_1 → gemini_2 → groq`, anthropic dormant
  (budget-safe).
- Device reaches `api.groq.com` (38 ms) and `generativelanguage.googleapis.com`
  (66 ms), 0% loss.

## Consequences

- ADB key management is now reliable regardless of whether the Keys screen is open.
- A `.bak_preinject` of `CteKeysActivity.kt` exists in the harness; delete after Mars
  confirms the on-device behaviour.
- If only `GEMINI_KEY_1` was set (not `_2`), the `gemini_2` rate-limit fallback has
  no key and falls through to `groq` — still covered.

## Open

- Live end-to-end (type a trigger in a focused field → provider response inserted)
  not exercised from host; all prerequisites are green. Confirm on-device when convenient.
- Leftover throwaway KeyVault entries from testing (`__TEST_INJECT_KEY__`,
  `__OVERWRITE_TEST__`, `__NEWINTENT_TEST__`) are harmless non-provider refs; reset
  the prefs file while no real keys matter, or leave them (never shown in UI).
