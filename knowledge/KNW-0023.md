# Cowork Desktop Capabilities — Verified 2026-05-25

## The Two Execution Environments

| Environment | Tool | What It Is | Filesystem |
|-------------|------|-----------|------------|
| **Linux sandbox** | `mcp__workspace__bash` | Isolated container. Separate OS, separate filesystem. Cannot see USB devices, Windows drivers, or local hardware. | `/sessions/.../mnt/` mount points only |
| **Windows host** | `mcp__Windows-MCP__PowerShell` | **Runs directly on DESKTOP-SH8JARJ.** Full PowerShell on the real machine. Same user context as Mars. Sees USB, drivers, ADB, network, processes, PnP devices — everything. | `C:\Users\MarsBase\...` real paths |

## What Windows MCP PowerShell CAN Do (verified this session)

- **ADB commands** — `adb devices`, `adb shell`, `adb connect`, `adb tcpip`, `adb push/pull` — all work
- **USB driver rebind** — `Disable-PnpDevice` / `Enable-PnpDevice` — replaces physical unplug
- **Process management** — `Start-Process`, `taskkill`, `Get-Process` — launch/kill scrcpy, adb, anything
- **PnP device enumeration** — `Get-PnpDevice` — see all USB hardware
- **File system** — full read/write to `C:\Users\MarsBase\*`
- **Network** — `adb connect` to WiFi devices, network testing
- **Settings changes on phone** — `adb shell settings put` — modify Android system settings
- **Script execution** — run .bat, .ps1, any executable
- **Shell commands on phone** — `adb shell` executes anything on the Android side

## What DOESN'T Apply Here

| Common Sandbox Limitation | Applies? | Why Not |
|--------------------------|----------|---------|
| "Can't access USB devices" | **NO** | Windows MCP runs on the host. ADB sees `R5CN81CDXJV` directly. |
| "Can't run local executables" | **NO** | Can launch scrcpy.exe, adb.exe, any .bat/.ps1 |
| "Can't modify hardware state" | **NO** | PnP driver rebind proven working — disables/enables USB devices |
| "Can't access local network" | **NO** | `adb connect 192.168.1.105:5555` works |
| "Isolated filesystem" | **NO** | Full access to `C:\Users\MarsBase\*` via PowerShell |
| "Can't kill/start processes" | **NO** | `taskkill /f /im scrcpy.exe` and `Start-Process` both work |
| "Read-only access" | **NO** | Write access to filesystem, can push files to phone via ADB |

## What I Actually CAN'T Do

| Limitation | Reason | Workaround |
|-----------|--------|------------|
| **Type into Terminal/PowerShell windows** | Computer-use grants Terminal at "click" tier — no keyboard input | Use `mcp__Windows-MCP__PowerShell` tool instead (runs commands directly) |
| **Type into VS Code / IDEs** | Same "click" tier restriction | Use Read/Write/Edit file tools |
| **Click inside browser windows** | Computer-use grants browsers at "read" tier | Use `mcp__Claude_in_Chrome__*` tools |
| **See scrcpy window in screenshots** | scrcpy.exe not in computer-use allowlist + user denied access request | Mars can see it — I just can't verify visually |
| **Run commands that take >45s** | `mcp__workspace__bash` has 45s timeout; `mcp__Windows-MCP__PowerShell` has 30s default | Break long operations into steps; use `Start-Process` for detached launches |
| **Admin/elevated operations** | UAC prompts can't be clicked | Ask Mars to approve UAC if needed |

## Rule: Default to Windows MCP

When Mars asks to run a command, execute a script, interact with ADB, or control hardware:
1. **Use `mcp__Windows-MCP__PowerShell` first** — it runs on the real machine
2. **Never say "I'm in a sandbox"** — that's the bash tool, not PowerShell
3. **Never ask Mars to paste commands** — run them directly
4. **If PowerShell times out** — the command is probably still running (e.g., scrcpy). Use `Start-Process` for long-running programs.
