# Context Compiler — Scoring Algorithm

## Purpose

The context-compiler plugin produces a token-budgeted briefing (`_system/_briefing.md`) that gives an AI session the most relevant vault context within a fixed token window. It solves the core problem of cross-session continuity: each new Claude session starts with no memory, so the briefing reconstructs working context from vault state.

## Scoring Dimensions

Every note in the vault receives a composite score from four weighted dimensions:

### 1. Recency (weight: 0.3)

- **Input:** File modification time (mtime from filesystem)
- **Scale:** Linear 0 to 1, where 1.0 = modified right now, 0.0 = oldest file in vault
- **Rationale:** Recently touched notes are more likely to be relevant to the current work stream.

### 2. Change Delta (weight: 0.3)

- **Input:** Whether the file was modified since the last compile run
- **Scale:** 1.0 if modified since last compile, otherwise decays toward 0 over time
- **Rationale:** Notes that changed between sessions represent new information the AI hasn't seen. This is the strongest signal for "what's new."

### 3. Link Centrality (weight: 0.2)

- **Input:** Count of inbound links to the note (other notes linking to it via `[[wikilinks]]`)
- **Scale:** Normalized by the maximum inbound link count in the vault (0 to 1)
- **Rationale:** Heavily-linked notes are structural hubs — decisions that affect many things, knowledge that many notes depend on.

### 4. Importance (weight: 0.2)

- **Input:** Frontmatter `importance` field
- **Scale:** `critical` = 1.0, `high` = 0.75, `normal` = 0.5, `low` = 0.25
- **Rationale:** Explicit human/AI signal that some notes matter more regardless of recency or links.

## Composite Score

```
score = (recency * 0.3) + (change_delta * 0.3) + (link_centrality * 0.2) + (importance * 0.2)
```

All scores fall in [0, 1]. Notes are sorted descending by composite score.

## Token Budget and Greedy Packing

Default budget: **8,000 tokens**. The compiler walks the sorted note list and greedily includes notes until the budget is exhausted.

Token estimation per note uses:
1. The frontmatter `tokens` field if present (pre-calculated by the author)
2. Fallback: `content.length / 4` (rough character-to-token approximation)

Notes that would exceed the remaining budget are skipped; smaller notes further down the list may still fit.

## Output Structure

The compiled briefing at `_system/_briefing.md` contains these sections:

1. **Vault State** — Total notes, last compile timestamp, token budget used
2. **Changed Since Last Compile** — Notes with change_delta = 1.0, listed with summaries
3. **Key Context** — Highest-scoring notes that fit the budget, with frontmatter and truncated body
4. **Open Tasks** — Tasks where status != done, sorted by priority
5. **Recent Decisions** — Decisions from the last 7 days with status and chosen option
6. **Drift Warnings** — Summary of current drift report (if any high/critical items exist)

## Operational Notes

- The compile can be triggered manually via command palette or runs automatically on vault open.
- The briefing is the recommended starting context for any new AI session — paste or reference it first.
- Notes in `_system/` are excluded from scoring (they are outputs, not inputs).
