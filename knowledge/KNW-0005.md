# Drift Detection — Four Scanner Types

## Purpose

The drift-detector plugin continuously monitors vault integrity. "Drift" is any state where the vault's contents diverge from its intended structure or internal consistency. The plugin runs four independent scanners, each targeting a different failure mode.

## Severity Levels

All findings are classified into four levels:

- **Critical** — Data integrity issue; must be resolved immediately (e.g., duplicate IDs)
- **High** — Likely incorrect information being referenced; resolve soon (e.g., stale references > 7 days, references to superseded decisions)
- **Medium** — Potential issue worth reviewing (e.g., minor staleness, type mismatches, overlapping decision scope)
- **Low** — Housekeeping (e.g., orphaned notes)

## Scanner 1: Stale Reference Detection

**What it checks:** Notes with a `last_verified` date in their frontmatter. If any note linked from that note (via `[[wikilinks]]`) has been modified *after* the `last_verified` date, the reference is stale — the knowledge note may be asserting things about a target that has since changed.

**Severity rules:**
- Drift > 7 days between `last_verified` and linked note's mtime: **High**
- Drift <= 7 days: **Medium**

**Why it matters:** Knowledge entries (like this one) cache understanding of other vault contents. When those contents change, the cached understanding may be wrong. This scanner is the primary mechanism for keeping knowledge entries honest.

## Scanner 2: Orphan Detection

**What it checks:** Builds a complete inbound link map for the vault. Notes with zero inbound links that have not been modified in 14+ days are flagged as orphans.

**Exclusions:** Everything in `_system/` is excluded (auto-generated files are not expected to be linked to).

**Severity:** Always **Low**. Orphans are not errors — they may be new notes that haven't been integrated yet. But long-standing orphans suggest forgotten or abandoned content.

**Why it matters:** Orphans represent unreachable knowledge. They won't appear in link centrality scoring (see [[KNW-0003]]), won't be surfaced by Dataview queries that traverse links, and may silently go stale.

## Scanner 3: Schema Violation

**What it checks:** Validates every note's frontmatter against its type definition in `_schemas.yaml`.

**What it catches:**
- Missing required fields (e.g., a decision without `status`) — **High**
- Wrong field types (e.g., `importance` set to `"very high"` instead of one of the allowed enum values) — **Medium**
- Unknown note types (frontmatter `type` not defined in schema) — **Medium**

**Why it matters:** The entire automation layer depends on well-formed frontmatter. A decision without a `status` field will not appear in the decision index. A knowledge note with a misspelled `confidence` value will not sort correctly in the compiler.

## Scanner 4: Contradiction Detection

**What it checks:** Three distinct contradiction types:

### 4a. Duplicate IDs (Critical)
Scans all notes for their `id` frontmatter field. Any two notes sharing the same ID is a critical error — IDs are the primary key for the entire system. Links, supersession chains, and indexes all depend on ID uniqueness.

### 4b. Overlapping Affects (Medium)
For all decisions with `status: accepted`, compares their `affects` arrays. If two active decisions list the same affected area, they may conflict. This doesn't necessarily mean there's a problem, but it flags areas where human review is warranted.

### 4c. Superseded Decision References (High)
Scans knowledge notes for `[[DEC-NNNN]]` links. If the referenced decision has `status: superseded`, the knowledge note may be operating on outdated assumptions. The correct action is usually to update the knowledge note to reference the superseding decision and re-verify its content.

## Output

All findings are written to `_system/_drift_report.md`, grouped by severity (critical first, then high, medium, low). Each finding includes:

- The source note path
- The scanner that flagged it
- A human-readable description of the issue
- Suggested remediation

## Status Bar Indicator

The plugin displays a live indicator in Obsidian's status bar:

- Green check: No high or critical findings
- Yellow warning: Medium findings present
- Red alert: High or critical findings present

The indicator count reflects total open findings. Clicking it opens the drift report.

## Operational Notes

- Scanners run on vault open, on file save, and on a configurable interval (default: 5 minutes).
- Scanning is incremental where possible — only re-scanning notes whose mtime has changed since the last run.
- The drift report is excluded from context-compiler scoring (it's in `_system/`), but drift *warnings* are summarized in the compiled briefing's final section.
