# Decision Supersession Chains

## Decision Lifecycle

Every decision in the vault has a unique ID (`DEC-NNNN`) and a status field that tracks its lifecycle:

- `proposed` — Under consideration, not yet accepted
- `accepted` — Active and in effect
- `superseded` — Replaced by a newer decision
- `rejected` — Considered and explicitly declined

## How Supersession Works

When a decision needs to be revised or replaced:

1. A **new** decision note is created (e.g., `DEC-0012`).
2. The new note's frontmatter includes `supersedes: DEC-0005`, pointing to the decision it replaces.
3. The old decision's status is automatically flipped to `superseded` by the decision-ledger plugin.
4. The old decision remains in the vault as an immutable historical record.

This creates **traceable chains**: `DEC-0001` -> `DEC-0005` -> `DEC-0012`. Reading the chain shows how thinking evolved — what was originally decided, why it changed, and what replaced it.

## The "Supersede Decision" Command

The decision-ledger plugin provides a command palette action: **Supersede Decision**. When invoked on an existing decision:

1. Creates a new decision note from the decision template.
2. Pre-fills `supersedes:` with the current decision's ID.
3. Copies the original decision's **context** section into the new note (preserving the problem statement).
4. Copies the original **alternatives** list so all previously considered options are visible.
5. Adds the original **chosen** option to the alternatives with a note that it was the previous choice.
6. Leaves the new `chosen` and `rationale` fields empty for the author to fill.

This ensures no context is lost during supersession — the new decision inherits full awareness of what came before.

## Index Builder

The decision-ledger plugin maintains `_system/_decision_index.md`, which:

- Lists all decisions grouped by status (accepted, proposed, superseded, rejected)
- Visualizes supersession chains as linked sequences
- Shows which notes are `affected` by each active decision (via the `affects` frontmatter field)
- Updates automatically when decisions change

## Interaction with Drift Detection

The drift-detector (see [[KNW-0005]]) specifically watches for:

- **Knowledge notes referencing superseded decisions** — flagged as high severity because the knowledge may be based on outdated architectural choices
- **Overlapping `affects` fields** in accepted decisions — flagged as medium severity because two active decisions governing the same area may conflict

## Design Rationale

Decisions are never edited in place or deleted. The append-only, chain-based model ensures:

1. Full audit trail of architectural evolution
2. AI sessions can reconstruct decision history without ambiguity
3. No silent changes — every revision is a visible, linked event
4. Old reasoning is preserved even when conclusions change
