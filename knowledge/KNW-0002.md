# Vault Architecture

## Core Principle

Notes are **machine-readable first**. Frontmatter is the primary structured data; the Markdown body provides supplementary context for human readers and for AI sessions that need deeper understanding. Every note type has a defined schema, and every automation layer reads frontmatter fields as its data source.

## Schema System

`_schemas.yaml` is the single source of truth for note structure. It defines typed frontmatter for each note type:

| Type | Key Fields | Purpose |
|------|-----------|---------|
| `decision` | id, status, chosen, alternatives, supersedes, affects | Architectural and workflow decisions with full audit trail |
| `session` | id, date, goals, outcomes | Per-conversation session logs |
| `knowledge` | id, subject, confidence, last_verified, source | Cached understanding for cross-session reuse |
| `task` | id, status, priority, blocked_by | Trackable work items |
| `briefing` | id, compiled_at, token_count | Auto-generated context summaries |

Required fields, allowed values, and types are all enforced — the drift-detector plugin (see [[KNW-0005]]) flags schema violations automatically.

## Folder Structure

Flat, one-level deep, semantic naming. No nesting. This was an explicit architectural decision (DEC-0001). Folders include:

- `decisions/` — DEC-NNNN files
- `sessions/` — SESSION-NNNN files
- `knowledge/` — KNW-NNNN files (this folder)
- `tasks/` — TASK-NNNN files
- `_templates/` — Templater folder templates, one per note type
- `_system/` — Auto-generated files (briefings, drift reports, indexes)

## Template System

Templater plugin with **folder templates** — dropping a new file into `decisions/` automatically applies the decision template, pre-filling frontmatter with the correct schema and auto-incrementing the ID. This eliminates manual schema adherence.

## Dashboard

`Dashboard.md` at vault root uses Dataview queries to surface live vault state:

- Open tasks by priority
- Recent decisions and their status
- Knowledge entries approaching staleness (last_verified age)
- Drift report summary
- Session history

## Automation Layer — Three Custom Plugins

1. **context-compiler** — Scores and packs notes into a token-budgeted briefing for AI sessions. See [[KNW-0003]].
2. **decision-ledger** — Manages decision lifecycle: creation, supersession chains, index building. See [[KNW-0004]].
3. **drift-detector** — Runs four scanner types to catch staleness, orphans, schema violations, and contradictions. See [[KNW-0005]].

These plugins operate on the structured frontmatter, making the schema the contract between human authoring and automated maintenance.
